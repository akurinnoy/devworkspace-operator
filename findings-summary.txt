8. IMPLEMENTATION GAPS (for T2 and T3)

   a) EnsureHomeInitContainerFields NEVER returns an error.
      - The function signature returns error, but the body always returns nil.
      - If T2/T3 want to add command validation (e.g., reject disallowed commands), the error
        path must be implemented here. Currently any user-supplied command is accepted silently.

   b) No test for EnsureHomeInitContainerFields.
      - custom_init_test.go and persistentHome_test.go have no test for this function.
      - T2 should add a test in pkg/library/home/ covering:
          * container with empty Command → gets default ["/bin/sh", "-c"]
          * container with non-empty Command → Command is preserved
          * VolumeMounts are always set to [{Name: "persistent-home", MountPath: "/home/user/"}]

   c) MergeInitContainers is fully tested (merge_test.go, lines 25-144).
      The "preserve user-configured init-persistent-home content" test (line 98) confirms
      that user-supplied Image, Args, and Env override defaults while Command is preserved
      from the base if not in the patch.

   d) Command is set at Devfile level (inferInitContainer, line 238-248):
      `Command: []string{"/bin/sh", "-c"}` and `Args: []string{initScript}`
      This is the default init-persistent-home container added to the Devfile TemplateSpec.
      The Kubernetes-level init container from DWOC config (WorkspaceConfig.InitContainers)
      is handled separately and merged afterward. These are two separate paths.

   e) No validation that a user-supplied Args value is non-empty or valid.
      EnsureHomeInitContainerFields does NOT set a default Args value.
      A user could supply an init-persistent-home container in DWOC config with no Args,
      which would result in no script being run.

   f) The controller guard at line 405 requires PersistUserHome to be set (not nil)
      for the DisableInitContainer check at line 407, but there is no nil-guard for
      workspace.Config.Workspace.PersistUserHome before dereferencing. If PersistUserHome
      is nil, pointer.BoolDeref will panic. This is a latent nil-dereference risk at line 407.
